## Ключевые принципы дизайна и архитектуры Lift

### 1) Архитектурная модель

- **Clean Architecture + DDD** как базовый подход.
- Слои строго разделены: **Domain → Application → Infrastructure → Presentation**.
- Зависимости направляются внутрь (в сторону Domain), а не наоборот.

### 2) Организация кода

- **Vertical Slice / Feature-first**: код группируется по фичам (`tasks`, `today`, `stats`, `logs`, `onboarding`).
- Общие абстракции и инфраструктура живут в `shared`.

### 3) Презентационный слой

- Паттерн **MVVM**:
  - **View** (React-компоненты): только отображение и UI-события.
  - **ViewModel** (Zustand): состояние экрана и оркестрация UI-логики.
  - **Model/Application**: use cases и доменная логика.

### 4) Домен и бизнес-логика

- Бизнес-правила размещаются в **Domain/Application**, а не в UI.
- Используются **Entities**, **Value Objects** (`TaskId`, `DateOnly`, `NonEmptyTitle`) и **Domain Events**.
- Use case — единственная точка выполнения пользовательского сценария.

### 5) Данные и синхронизация

- **Local-first**: локальная БД (IndexedDB через Dexie) — первичный источник данных для UX.
- Supabase используется для синхронизации и real-time сценариев.
- Репозитории скрывают детали хранения/сети за интерфейсами.

### 6) Событийная модель

- Гарантии: **at-least-once**, порядок событий в рамках одного aggregate, retry/backoff, durability, dead-letter.
- Обработчики событий должны быть **идемпотентными**.
- Обработка начинается после фиксации транзакции (post-commit).

### 7) Качество и сопровождаемость

- Строгая типизация TypeScript, DI через TSyringe.
- Тесты на уровнях unit/integration/e2e.
- При изменениях сохранять границы слоев, не допускать утечек инфраструктурных деталей в домен/презентацию.
